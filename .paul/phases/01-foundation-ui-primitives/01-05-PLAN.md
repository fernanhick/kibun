## Goal

If Lottie is not installed, configured, and validated against real assets before Phase 2 begins, Phase 2 cannot build or test any screen. This plan closes that dependency.

  <name>Task 3: Create Shiba component</name>
  <files>
    src/components/Shiba.tsx
  </files>
  <action>
    Create `src/components/Shiba.tsx`:

    **Type definitions:**
    ```typescript
    export type ShibaVariant = 'happy' | 'excited' | 'sad' | 'neutral';

    interface ShibaProps {
      variant: ShibaVariant;
      size?: number;        // Diameter in pt — default 120
      loop?: boolean;       // default true
      autoPlay?: boolean;   // default true
      onFinish?: () => void;
      style?: StyleProp<ViewStyle>;
    }
    ```

    **Animation map:**
    ```typescript
    // No explicit type annotation — require() returns any, which is directly assignable to
    // LottieView's source prop (typed as AnimationObject in lottie-react-native v7).
    // Annotating as Record<ShibaVariant, object> would produce a TypeScript strict-mode
    // error: object is not assignable to AnimationObject (Record<string, unknown>) because
    // object lacks an index signature. Let any flow through instead.
    const ANIMATIONS = {
      happy:   require('../assets/lottie/shiba-happy.json'),
      excited: require('../assets/lottie/shiba-excited.json'),
      sad:     require('../assets/lottie/shiba-sad.json'),
      neutral: require('../assets/lottie/shiba-neutral.json'),
    };
    ```
    Note: Paths are relative to `src/components/` — `'../assets/lottie/...'` is correct.
    JSON requires are resolved by Metro bundler at build time; no @assets alias needed.
    TypeScript infers `{ happy: any, excited: any, sad: any, neutral: any }` — indexing with
    `ShibaVariant` (union of the four literal keys) works in TypeScript 5.x.

    **Component:**
    ```tsx
    export function Shiba({
      variant,
      size = 120,
      loop = true,
      autoPlay = true,
      onFinish,
      style,
    }: ShibaProps) {
      return (
        // Wrap LottieView in View for accessibility props. LottieViewProps does not
        // guarantee accessibilityLabel / accessibilityRole in its TypeScript interface —
        // passing them directly to LottieView would be a strict-mode compile error if they
        // are not in the declared props. View always accepts all accessibility props.
        <View
          style={[{ width: size, height: size }, style]}
          accessibilityLabel={`Shiba ${variant}`}
          accessibilityRole="image"
        >
          <LottieView
            source={ANIMATIONS[variant]}
            autoPlay={autoPlay}
            loop={loop}
            onAnimationFinish={onFinish}
            style={StyleSheet.absoluteFill}
          />
        </View>
      );
    }
    ```

    **Imports required:**
    - `LottieView` from `lottie-react-native`
    - `View`, `StyleSheet`, `StyleProp`, `ViewStyle` from `react-native`

    Avoid: Importing from `react-native-reanimated` directly in this file — Shiba uses
    `lottie-react-native`'s LottieView which handles Reanimated internally.
    Avoid: Putting accessibility props directly on LottieView — use the View wrapper above.
    Avoid: Explicit Record<ShibaVariant, object> annotation on ANIMATIONS — see comment above.
  </action>

  <action>
    Create `src/components/SplashScreenView.tsx`:

    A full-screen view that displays the Shiba happy animation centered with the app name.
    Used in `app/_layout.tsx` during auth initialization. Also available for Phase 2 onboarding
    entry animations.

    **Props interface:**
    ```typescript
    interface SplashScreenViewProps {
      onFinish: () => void;
    }
    ```
    `onFinish` is required — it signals to `_layout.tsx` that the animation has completed,
    which is the trigger for calling `SplashScreen.hideAsync()` and rendering the main Stack.
    Without this callback, `_layout.tsx` has no way to know the animation played and the
    SplashScreenView would never be visible (see Task 5 rationale for full timing explanation).

    **Component:**
    ```tsx
    export function SplashScreenView({ onFinish }: SplashScreenViewProps) {
      return (
        <View style={styles.container}>
          <Shiba
            variant="happy"
            size={160}
            loop={false}
            autoPlay
            onFinish={onFinish}
          />
          <Text style={styles.title}>kibun</Text>
          <Text style={styles.subtitle}>気分</Text>
        </View>
      );
    }
    ```
    Note: `loop={false}` — the animation plays exactly once, then `onFinish` fires. This
    guarantees the user sees the full Shiba animation before the main app renders. If the
    Shiba animation were `loop={true}`, the `onAnimationFinish` callback would fire after
    every loop and the `setSplashDone` logic in _layout.tsx would fire on the first loop
    completion (this still works, but `loop={false}` is semantically cleaner: the splash
    plays once and is done).

    **StyleSheet:**
    - `container`: `flex: 1`, `alignItems: 'center'`, `justifyContent: 'center'`,
      `backgroundColor: colors.background`
    - `title`: `fontSize: typography.sizes.display`, `fontWeight: typography.weights.bold`,
      `color: colors.text`, `marginTop: spacing.md`
    - `subtitle`: `fontSize: typography.sizes.xl`, `fontWeight: typography.weights.regular`,
      `color: colors.textSecondary`, `marginTop: spacing.xs`

    **Imports required:**
    - `View`, `Text`, `StyleSheet` from `react-native`
    - `Shiba` from `./Shiba`
    - `colors`, `typography`, `spacing` from `@constants/theme`

    Avoid: SafeAreaView in SplashScreenView — this screen does not have tab bars or navigation
    headers; full-bleed centered layout is correct. SafeAreaProvider wraps the main Stack in
    _layout.tsx but SplashScreenView renders outside that tree.
    Avoid: Hardcoded color values — use tokens only.
    Avoid: `loop={true}` with `onFinish` — use `loop={false}` so onFinish fires once after a
    complete single play, not after every repetition.
  </action>

  <name>Task 5: Wire SplashScreenView into _layout.tsx + update barrel</name>
  <files>
    app/_layout.tsx,

  </files>
  <action>
    **1. Update `app/_layout.tsx`:**
    Three changes are required: new import, new `splashDone` state, updated `useEffect`, and
    updated null guard. This replaces the existing `useEffect` for `hideAsync`.

    Add import at the top of the file:
    ```typescript
    import { SplashScreenView } from '@components/SplashScreenView';
    ```

    Update the `React` import to include `useState`:
    ```typescript
    import React, { useState, useEffect } from 'react';
    ```

    Replace the existing `useEffect([isReady])` and null guard with:
    ```typescript
    export default function RootLayout() {
      const { isReady } = useAuth();
      const [splashDone, setSplashDone] = useState(false);

      // Delay native splash hide until BOTH auth resolves AND animation has played once.
      // Without splashDone, hideAsync fires when isReady=true, React immediately re-renders
      // to Stack — the native splash fades revealing Stack, not SplashScreenView. The Shiba
      // animation is never visible and AC-4 / Lottie validation cannot be verified.
      useEffect(() => {
        if (isReady && splashDone) {
          SplashScreen.hideAsync();
        }
      }, [isReady, splashDone]);

      // Show SplashScreenView until both conditions are met:
      // - isReady: auth resolved (prevents premature Stack render)
      // - splashDone: animation played once (ensures Shiba is visible to user)
      if (!isReady || !splashDone) {
        return <SplashScreenView onFinish={() => setSplashDone(true)} />;
      }

      return (
        <SafeAreaProvider>
          <ErrorBoundary>
            <Stack initialRouteName="(tabs)">
              <Stack.Screen name="(tabs)" options={{ headerShown: false }} />
            </Stack>
          </ErrorBoundary>
        </SafeAreaProvider>
      );
    }
    ```

    **Timing guarantee this creates:**
    1. App starts → SplashScreenView renders → Shiba animation begins (behind native splash)
    2. Auth resolves → `isReady=true` → effect runs: `splashDone=false` → hideAsync NOT called
    3. Animation finishes → `setSplashDone(true)` → effect runs: both true → `hideAsync()` called
    4. React re-renders: `!isReady || !splashDone` = false → Stack renders
    5. Native splash fades (async) while Stack renders underneath
    6. User sees: native splash → Shiba animation (fully visible during step 3) → main tabs ✓

    If auth is slower than the animation (rare but possible on cold starts):
    - Animation finishes → `splashDone=true`, `isReady=false` → last frame frozen, user waits
    - Auth resolves → `isReady=true` → effect fires hideAsync → Stack renders
    - Acceptable UX: user sees loading state rather than a blank screen

    **2. Update `src/components/index.ts`:**
    Add Shiba and SplashScreenView exports:
    ```typescript
    export { Button } from './Button';
    export { Card } from './Card';
    export { MoodBubble } from './MoodBubble';
    export { Screen } from './Screen';
    export { Shiba } from './Shiba';
    export type { ShibaVariant } from './Shiba';
    export { SplashScreenView } from './SplashScreenView';
    ```

    Avoid: Changing the SafeAreaProvider or ErrorBoundary structure in _layout.tsx —
    SplashScreenView intentionally renders outside them (it's a pre-auth loading state)
  </action>
